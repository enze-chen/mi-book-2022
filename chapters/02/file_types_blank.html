Text files¬∂

Back to top

From yesterday‚Äôs exercises and your own experience, you‚Äôve worked with text files in many contexts, which speaks to their great flexibility. Indeed this is one of their strengths, as they simply consist of characters on lines that can then be represented in Python as a sequence of strings.

To revisit an example from yesterday, let‚Äôs open the file mentors.txt and print its contents.

Text files are great, because we can store all sorts of information in them and they are human readable. When we talk about readability (interpretability), we typically refer to humans; but in MI, you‚Äôll soon see that there seems to be an unfortunate tradeoff between what‚Äôs interpretable for humans and what‚Äôs interpretable for computers.

Another quick note about text files is that they don‚Äôt have to end in .txt to be a ‚Äútext file.‚Äù A Python file (.py extension) or Jupyter notebook file (.ipynb), for example, are considered text files too! In Module 3, when you‚Äôre working with DFT calculations, you‚Äôll likely see a lot of text files that don‚Äôt even have an extension! (This is possible on any OS, though rarely helpful on ones with GUIs.)

When the file is structured¬∂

A more subtle answer to the above question would be when the file is still a text file, but the text is written in a way that has some pre-defined, internal structure. The flexibility of text files is really nice if we‚Äôre writing prose or poetry, but data are usually more systematically organized to facilitate access. This means there might be a more specialized/targeted way to read in the contents of the file than going line by line and interpreting the contents as strings, particularly as a lot of data are numeric. As we move forward and discuss different examples of structured data, we invite you to think about what we said earlier about the tradeoff between human interpretability and computer interpretability.

Writing JSON files¬∂

If you have some hierarchical data stored inside lists and dictionaries, you may want to save it to an external JSON file. As a very simple example, let‚Äôs take the list of band gap values you created in the previous exercise and try writing that list (band_gaps) to a JSON (for any other data, the code is the same, and only the file content structure/hierarchy will be different). We use the json.dump() function as follows:

with open('path/to/output_file.json', 'w') as f:   # note the 'w' for write!
    json.dump(data_object, f, indent=4)            # indent=4 is a best practice

When applied to our data, the result is:

with open('../../assets/data/band_gaps_only.json', 'w') as f:
    json.dump(band_gaps, f, indent=4)
